Methodology

What is a signal? Whenever the model's probability differs from the market price by more than the user's edge threshold (default 8pp), we record a signal in weather_signals_log. Each signal has a market id, a question, a model probability, the yes-price at signal time, and a UTC timestamp.

How is a signal resolved? After the market's target date passes, we fetch the observed daily-max temperature from Open-Meteo's archive API and parse the market question's threshold (above X°F, below X°F, between X and Y, …). The resolution writes to weather_resolutions. This pairing runs hourly so today's markets are usually scored by 03:00 UTC the next day.

What does the daily rollup include? Aggregate counts, win rate against the sign of the edge, Brier score, log loss, and a 10-bucket reliability diagram for that day. The per-day JSON is canonical (sorted keys, deterministic separators) so the SHA-256 hash is reproducible.

How is the chain signed? Each row's HMAC-SHA256 is over {date, payload, content_hash, prev_hash} using a key derived from the server's secret (domain-separated from the credential vault). The chain links via prev_hash; editing a past day breaks every subsequent signature.

How can I verify? /api/public/track-record/verify walks the chain top-to-bottom and reports the first inconsistency. The latest content hash is also visible above — operators can publish that single hash to a timestamping service (e.g. Twitter, a Git commit, OpenTimestamps) to commit to the entire history.
