Everything under /api/public/v1/* is the stable developer surface. The older /api/v1/* paths continue to work for historical clients but new integrations should point here.

1. Authentication

Every request carries a Bearer token:

Authorization: Bearer narve_<your-api-key>

Mint keys at /settings/api-keys. Each key is displayed once at creation — we store only the SHA-256 hash. If you lose a key, revoke it and mint a new one.

Keys carry one or both scopes:

  • read — all GET endpoints. Granted automatically to every key.
  • write — POST endpoints (currently POST /predictions). Request on key creation; granted automatically on Pro / Enterprise tiers.

2. Rate limits

Per-key, measured in requests per clock hour (UTC). Resets at the top of every hour.

  • Free / Trader tier — 1 key, 1,000 req/hr
  • Pro tier — 5 keys, 10,000 req/hr each
  • Enterprise — custom, contact us

When you exceed a limit you get a 429 with Retry-After in seconds and X-RateLimit-* headers telling you where you stand. Your current hour's usage is also queryable at GET /api/public/v1/usage.

3. Endpoints

GET/markets

Search live markets. Query params: q (search term), limit (max 200).

curl -H "Authorization: Bearer narve_$KEY" \
  "https://narve.ai/api/public/v1/markets?q=election&limit=20"

{"markets": [...], "limit": 20, "q": "election"}

GET/markets/{slug}

One market by slug. Returns the latest snapshot.

curl -H "Authorization: Bearer narve_$KEY" \
  https://narve.ai/api/public/v1/markets/us-election-2028

GET/markets/{slug}/predictions

All predictions narve has extracted for this market. Query: limit (max 500).

GET/markets/{slug}/history

Time-series of market price + narve credibility-weighted probability. Query: limit (max 2000).

GET/sources

Credibility-scored sources. Query: limit (max 500), offset.

{"sources": [{"source_handle": "nate_silver", "global_credibility": 0.68,
  "accuracy_unlocked": true, "total_predictions": 412, ...}],
 "total": 1843, "limit": 100, "offset": 0}

GET/sources/{handle}

Full profile for one source, including calibration + category breakdown.

GET/sources/{handle}/predictions

Every tracked prediction for a source, newest first. Query: limit (max 500).

GET/sources/{handle}/history

Credibility snapshots over time. Useful for charting.

GET/feed

Recent extracted predictions across all sources. Query: limit (max 500), category.

GET/best-bets

Current top edge opportunities — narve-probability vs market-price delta. Query: limit (max 100).

GET/calendar

Upcoming market resolutions with deadlines.

POST/predictions — requires write scope

Submit your own prediction on any market. Body is JSON.

curl -X POST \
  -H "Authorization: Bearer narve_$KEY" \
  -H "Content-Type: application/json" \
  -d '{"market_slug":"btc-gt-100k-eoy",
       "predicted_outcome":"YES",
       "predicted_probability":0.72,
       "reasoning":"Mining hashrate + spot ETF flows…",
       "is_public":true}' \
  https://narve.ai/api/public/v1/predictions

{"prediction_id": 19281, "status": "ok"}
Direction (YES/NO) is locked at submission. You can edit probability / reasoning for 24 hours from your browser at /predictions.

GET/usage

Your current hour's request count and reset time. No params.

{"key_id": 42, "key_prefix": "narve_AbC12", "name": "backtest bot",
 "scopes": ["read", "write"], "rate_limit_hour": 10000,
 "requests_this_hour": 1247, "hour_bucket_start": 1714928400,
 "bucket_resets_at": 1714932000}

4. Error codes

StatusMeaningAction
400Bad request (invalid params / body)Check the error detail; fix the payload.
401Missing, malformed, or revoked API keyMint a new key at /settings/api-keys.
402Your plan doesn't include API accessUpgrade at /pricing.
403Key lacks required scope (e.g. write)Re-create the key with the right scope.
404Market / source / prediction not foundCheck the slug / handle.
409Conflict (e.g. duplicate active prediction)Wait for resolution or edit the existing row.
429Hourly rate limit exceededHonour Retry-After; shard across multiple keys if you have a higher tier.

5. Webhooks

Register endpoints to receive push events instead of polling. Configure at /settings/webhooks.

Event types

  • best_bet.new — a new high-edge opportunity appears
  • market.resolved — a tracked market resolves
  • source.credibility_updated — a source's credibility score changes
  • insider_signal.new — a high-conviction source posts a new prediction
  • user.prediction.resolved — one of your own predictions resolves

Payload + signature

Every delivery is an HTTP POST of JSON. narve computes HMAC-SHA256(secret, raw_body) and sends it as X-Narve-Signature: sha256=<hex>. Verify server-side:

import hmac, hashlib

def verify(raw_body: bytes, signature: str, secret: str) -> bool:
    expected = "sha256=" + hmac.new(
        secret.encode(), raw_body, hashlib.sha256
    ).hexdigest()
    return hmac.compare_digest(expected, signature)

Retries

Non-2xx responses are retried 5 times with exponential backoff: 30s, 5m, 30m, 1h, 4h. Five consecutive failed deliveries (across any event) auto-disable the subscription and email you.

Test firing

The webhooks settings page has a Test button that sends a synthetic test.ping event so you can verify your endpoint before wiring up real events.
