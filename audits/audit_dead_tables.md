# Audit: Dead Tables in `gateway/auth.db`

Generated by `audits/_audit_dead_tables.py` (one-shot; not committed).

## Method

For each table in `gateway/auth.db`:

1. Enumerated all tables with `SELECT name FROM sqlite_master WHERE type='table'`.
2. Searched every `.py`, `.sql`, `.html`, `.js`, `.ts`, `.jsx`, `.tsx` file under `gateway/`
   (excluding `migrations/`, build artefacts, caches, backups, logs).
3. Classified each table as:
   - `LIVE_RW` — read and written in app code
   - `READ_ONLY` — read but never written in app code
   - `WRITE_ONLY` — written but never read in app code
   - `DEAD` — no reads or writes in app code (only present in schema/migrations or absent entirely)
   - `FTS_SHADOW` — auto-maintained FTS5 shadow table; status follows parent `*_fts`
   - `DEAD_FTS_SHADOW` — FTS shadow whose parent FTS table is dead

Read regexes look for `FROM <t>`, `JOIN <t>`.
Write regexes look for `INSERT INTO <t>`, `REPLACE INTO <t>`, `UPDATE <t> SET`,
`DELETE FROM <t>`, `CREATE TABLE <t>`, `ALTER TABLE <t>`, `DROP TABLE <t>`.

Migration files are excluded from the count — they declare schema but do not constitute live usage.

## Notes & Caveats

- Dynamic SQL (e.g. table name built at runtime via f-strings) may be missed by static regex search.
- Tables that exist only to back foreign keys for another table will still appear dead unless they
  themselves are read or written.
- FTS5 shadow tables (`*_fts_config`, `*_fts_data`, `*_fts_docsize`, `*_fts_idx`) are auto-maintained
  by SQLite and not directly read/written in code — they are flagged separately.
- Schema / declaration in `gateway/db.py` is treated as app code, so a table referenced only in
  `db.py` schema setup with no SELECT/INSERT/UPDATE/DELETE elsewhere will still be flagged dead.
