## Gaps / Findings

### G1 — No dedicated maskable safe-zone audit done on this pass
The maskable icons exist (`icon-maskable-192.png`, `icon-maskable-512.png`) and are sized correctly, but a manifest-level audit can't verify the inner 80% safe zone is honoured. If the maskable assets were generated by simply copying the "any" icon, Android launchers will crop the logo. Open the maskable PNGs in a maskable-icon previewer (e.g. maskable.app) to confirm — out of scope for this file-level audit but worth flagging.

### G2 — No `apple-touch-icon` rel link in manifest (handled elsewhere?)
The manifest itself doesn't need this (it's a `<link>` concern), but iOS still reads `apple-touch-icon` separately. `pwa_middleware.py` injects `apple-mobile-web-app-*` meta tags but a quick grep shows no `<link rel="apple-touch-icon">` declaration in the PWA head block (lines 107-130). iOS will fall back to a screenshot of the page for the home-screen icon. Low severity, but a clean PWA usually has both a 180×180 apple-touch-icon and the manifest icons.

### G3 — Missing `id` field
Manifest does not declare an `id`. Without it, Chrome derives the app identity from `start_url`, which means changing `start_url` later (e.g. away from `/dashboards`) would orphan installed PWAs and they'd reinstall as a new app. Adding `"id": "/"` or `"id": "narve.ai"` would future-proof the install identity. Low severity — only matters if start_url ever changes.

### G4 — No screenshots array
Manifest has no `screenshots` field. Chrome on Android shows a richer install prompt ("rich install UI") when 1+ screenshots with `"form_factor": "wide"` or `"narrow"` are declared. Without them, users get the minimal install banner. Cosmetic — install rate is the only thing affected.

### G5 — `prefer_related_applications` not set
Not declared, so defaults to `false` (correct behaviour — we want the PWA, not a Play Store redirect). Listing this explicitly as `false` is best practice but not required. Trivially low.

### G6 — Icon `purpose: "any maskable"` not used (split icons instead)
The manifest uses separate icons for `purpose: "any"` and `purpose: "maskable"` (4 files total) rather than dual-purpose `"any maskable"` icons. This is the safer pattern (no compromise on either rendering) and matches W3C guidance — not a gap, just noting that the design choice is intentional.

## Top 3 Priorities

1. **G1 — Verify maskable safe zone.** Visually confirm `icon-maskable-192.png` / `icon-maskable-512.png` have the logo inside the inner 80% circle. If not, Android adaptive icons will clip. Highest user-visible impact of anything found here.
2. **G2 — Add `<link rel="apple-touch-icon" href="/_gateway_static/img/icon-192.png">` (or a dedicated 180×180) to `pwa_middleware.py`'s `_PWA_HEAD`.** iOS home-screen icon quality is currently relying on whatever Safari guesses.
3. **G3 — Add `"id": "/"` to the manifest.** Cheap one-line change that locks the PWA install identity so future `start_url` tweaks don't orphan installed apps.
