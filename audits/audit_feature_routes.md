# Audit — feature-flag admin surface (`gateway/admin_routes.py`)

Adversarial review focused on: admin-only toggle gating, flag-key allowlist (no arbitrary key creation), audit log of flag changes, default-deny on unknown flags.

**File audited:** the spec asked for `gateway/feature_routes.py`, but **no such file exists** in the repo. The feature-flag admin endpoints live in `gateway/admin_routes.py` (lines 296-593, register block at L3021-3044). Evaluation library is `gateway/features.py` (169 lines). DB layer is `gateway/queries/admin.py` L537-691. Audit catalog is `gateway/security/audit.py` (389 lines). The phantom-filename framing is itself a finding (#L8 below): the surface the audit was scoped to is co-mingled inside a 3000-line `admin_routes.py` so any reviewer looking for "feature routes" by filename will miss it.

Reviewed at `HEAD`, working tree of `/Users/shocakarel/Habbig` (branch `main`, no local diffs touched).

**Severity counts:** 1 CRITICAL · 3 HIGH · 4 MEDIUM · 4 LOW

Severity rubric (same as the rest of the `audits/` tree):
- **CRITICAL** — auth bypass, undetected privileged mutation, data exfiltration.
- **HIGH** — exploitable with authed admin or near-admin precondition; integrity of the gating story.
- **MEDIUM** — defense-in-depth gaps, footguns that turn into incidents under a bad refactor, log hygiene.
- **LOW** — code-quality with security implications.

## Top-12 findings

### CRITICAL

- **CRITICAL | admin_routes.py L471-479, L553-562, L571-577 vs security/audit.py L26-75 | Every feature-flag mutation is silently un-audited.** The handlers reference `_a.AuditAction.FEATURE_FLAG_CREATE`, `FEATURE_FLAG_UPDATE`, and `FEATURE_FLAG_DELETE` (L473, L557, L573). Those attributes **do not exist** on the `AuditAction` class in `gateway/security/audit.py` — `grep -n "FEATURE_FLAG" gateway/security/audit.py` returns nothing, and `ALL_ACTIONS` (L77) only enumerates `user.*`, `token.*`, `source.*`, `scraper.*`, `system.*`, `email.watermark_trace`, and `admin.*`. Accessing the missing constant raises `AttributeError`. That `AttributeError` is then **swallowed** by the surrounding `_audit()` helper at L78-93: `try: ... from security import audit as _a; _a.log_action(...) except Exception: pass`. Effect: **`db.insert_audit_log` is never called for any flag CRUD**, so the WORM "who flipped what when" trail the spec demands does not exist. The reviewer-facing form of this bug is doubly bad — a `grep "FEATURE_FLAG_CREATE" gateway/` looks like audit IS wired up, but at runtime no row is ever written. (Same bug exists for impersonation: `IMPERSONATION_START`/`IMPERSONATION_END` referenced at admin_routes.py L148, L176 are also absent from `AuditAction`, so impersonation audit is similarly silent — out of scope here but flag it on the parent ticket.) | **Fix:** add `FEATURE_FLAG_CREATE = "feature_flag.create"`, `FEATURE_FLAG_UPDATE = "feature_flag.update"`, `FEATURE_FLAG_DELETE = "feature_flag.delete"` (and `IMPERSONATION_START`/`IMPERSONATION_END`) to `class AuditAction`, mirror them in `ACTION_LABELS`, and add a unit test that asserts `db.query_audit_log(action="feature_flag.update")` returns one row after `flag_save` is invoked. The swallow-everything `except: pass` in `_audit()` should also `log.warning("audit emit failed action=%s err=%s", action, exc)` so the next time someone breaks the wiring it shows up in stderr instead of going silent for a year.

### HIGH

- **HIGH | admin_routes.py L448-454 | No flag-key allowlist. Any admin can mint an arbitrary flag key the application has never heard of.** `flag_create` validates the key only with `re.fullmatch(r"[a-z0-9_\-]{1,80}", key)` — a syntactic check, not a semantic one. There is no registry of "known flags the code actually reads" (no `FLAGS_ALLOWLIST` constant, no `features.KNOWN_KEYS`, no enum). An admin can POST `/admin/flags` with `key=secret_admin_unlock` or `key=disable_billing` and that row lands in the table. Downstream `features.is_feature_enabled("secret_admin_unlock", user)` returns `False` for any code path that never references it — so the row is inert *today* — but the spec explicitly asked for "flag-name allowlist (no arbitrary key creation)" and the current implementation grants every admin a 1-million-keyspace persistence primitive in the DB. The footgun materialises the moment any code does `is_feature_enabled(user.get("flag") or "default")` or pulls flag keys from form input — at that point an admin can pre-seed a row that flips behaviour for the targeted user_id only. Also: there is no `key.lower()` normalisation, but the regex pins lowercase so `Key` is rejected — OK. There is also no reserved-prefix check (e.g., admin can create `__internal__` or `_system_`). | **Fix:** define `features.KNOWN_FLAGS: frozenset[str]` at module level (or load from a `FLAG_REGISTRY` dict that also carries description + default tier). `flag_create` should `if key not in features.KNOWN_FLAGS: raise HTTPException(400, "Unknown flag key. Register in features.KNOWN_FLAGS first.")`. Treat this as a deploy-gated allowlist — adding a new flag is a code change, not a DB write. As a separate guard: add a `chk_key_format CHECK (key GLOB '[a-z0-9_-]*')` constraint to the `feature_flags` table so the regex bypass surface (Unicode lookalikes, mixed-case) is closed at the SQL layer.

- **HIGH | admin_routes.py L532-562 | `flag_save` never reads the pre-mutation row for the audit `before` payload.** Even after the CRITICAL above is fixed and audit constants exist, the handler at L556-562 only passes `after=audit_kwargs` to `_audit()`. The `before=` kwarg of `log_action` (audit.py L213) is left at `None`. So the audit row will record the *new* state but not the *old* state — anyone reading audit_log a month later cannot reconstruct the diff "flag X went from 50% rollout to 100%" without correlating against a separate `updated_at` archive that doesn't exist. The handler already fetches the row at L536 (`row = db.get_feature_flag(...)`) to do the 404 check, so capturing `before = features.flag_to_dict(row)` is one line. | **Fix:** at L536 replace `row = ...` with `prior = db.get_feature_flag(key, subproduct_key=subproduct_key)` followed by `before_state = features.flag_to_dict(prior) if prior else None`, then pass `before=before_state` into `_audit()` at L556. Same fix applies to `flag_delete` at L566-578 (currently captures neither before nor after; the deleted state vanishes).

- **HIGH | admin_routes.py L581-593 | `/api/flags/evaluate/{key}` lets any authenticated free-tier user enumerate the entire flag table and rollout state.** The handler requires `_current_user(request)` (good — gates against anonymous) but does **not** require admin. So a non-admin can GET `/api/flags/evaluate/<any_key>?subproduct=<any_slug>` and learn `{"enabled": true/false, "subproduct_key": "..."}` for every (key, subproduct) tuple in the database — including kill-switches, internal feature names, and per-subproduct overrides. Combined with the no-allowlist finding above, an admin who creates `key=ceo_only_unlock` exposes its existence to every free-tier user the moment they brute-force a key dictionary. The 404 vs `enabled:false` distinction (`is_feature_enabled` returns False both for missing rows AND for known-but-not-enabled-for-you rows) reduces leak via the API itself, **but the response includes `subproduct_key` echoed back from the query-string** — which only tells the caller "you asked about X". The bigger leak is the JSON 200 status: a 200 means "I evaluated something", confirming the endpoint is alive and the caller is authed. That's enumeration. There's no rate limit on this path either, so a malicious user can probe 80-char^36 keyspace at full throttle bounded only by IP-level limits. | **Fix:** either gate the endpoint behind `_require_admin_user(request)` (preferred — flag evaluation for non-admin code should happen in-process via `features.is_feature_enabled`, not via an HTTP round-trip) or add per-user-id rate limit `_is_rate_limited(f"flag_eval:{user['user_id']}", 30, 60)` AND restrict to keys present in `features.KNOWN_FLAGS`. Bonus: drop the `subproduct_key` echo from the response — the caller already knows what they asked for.

### MEDIUM

- **MEDIUM | admin_routes.py L383 | `bool(form.get("enabled_globally"))` evaluates "false", "0", "off" as True.** `_parse_flag_form` converts the global toggle with raw `bool(...)` rather than parsing the string. A hand-crafted POST with `enabled_globally=false` (the literal four-letter string) flips the flag ON. The admin UI uses an HTML checkbox so unchecked-state sends no key at all — fine in the happy path. But (a) anyone scripting via curl, (b) any future REST client, or (c) a copy-pasted-from-stack-overflow JS payload that does `body: new URLSearchParams({enabled_globally: false})` will silently set the flag to enabled-globally. Same bug doesn't apply to `rollout_percentage` because it's parsed via `int(...)`. | **Fix:** replace `bool(form.get("enabled_globally"))` with `(form.get("enabled_globally") or "").strip().lower() in ("1", "true", "yes", "on")`. Also reject the request with 400 if the value is non-empty and not in the recognised set — silent coercion is the footgun.

- **MEDIUM | admin_routes.py L364-392 + queries/admin.py L614-666 | `flag_save` always overwrites every column with form state — there is no PATCH-style partial update.** `_parse_flag_form` synthesises a full kwargs dict on every save (tiers, user-id allowlists, rollout, description) and passes every field non-None into `update_feature_flag`. The DB helper at queries/admin.py L639+ honours non-None values, so a partial form submission silently zeroes any unspecified field. Example exploit: admin A renders `/admin/flags/voters_beta` (which currently has `enabled_for_user_ids=[1,2,3]`). Admin B simultaneously POSTs the form having only ticked `enabled_globally`. The form serialises with `enabled_user_ids=""`, `_parse_flag_form` returns `enabled_for_user_ids=[]`, and admin B's submit wipes admin A's allowlist. There is no optimistic concurrency check (`updated_at` precondition) and no field-omitted-vs-empty distinction — the form simply cannot represent "leave this field as-is". Combined with the missing `before` audit (HIGH above), the wiped state is unrecoverable. | **Fix:** add a hidden `_updated_at_seen=<prev_ts>` field to the edit form. In `flag_save`, refuse the save if `row["updated_at"] != int(_updated_at_seen)` (HTTP 409 Conflict, "Flag was modified by someone else since you opened the editor"). For the partial-update concern, introduce a `kwargs = {k: v for k, v in kwargs.items() if k in form}` filter so fields not present in the form body are not sent to `update_feature_flag`.

- **MEDIUM | admin_routes.py L338-349 vs L587 | `_normalize_subproduct` silently coerces unknown slugs to `None` (i.e., "global") instead of rejecting them.** When an admin POSTs `?subproduct=evil_slug`, `_normalize_subproduct` returns `None` because `evil_slug not in _subproduct_slugs()`. The handler then operates on the **global** row — silently. So a tampered URL becomes a global flag mutation. The intent (`# Coerce ... falls back to None (global)`) is documented as a feature, but it means an admin who *thinks* they're editing the `voters` override is actually editing the global row whenever they typo the slug. There's no UI feedback distinguishing "operated on subproduct X" vs "operated on global". And combined with the missing audit `before` state (HIGH above), a typo can flip global rollout without leaving any "you did this on global, not subproduct" breadcrumb. | **Fix:** when `raw` is non-empty but unknown, `raise HTTPException(400, f"Unknown subproduct slug: {raw}")` instead of returning `None`. Preserves the "empty string = global" UX path while killing the silent-coercion footgun.

- **MEDIUM | features.py L81-114 + queries/admin.py L685-691 | `record_feature_flag_event` writes one row per evaluation when `record_event=True`, unbounded.** The `feature_flag_events` table has no retention policy, no daily-aggregate index, no rate limit, and no caller-level guard against `record_event=True` being passed from a hot path (e.g., a per-request middleware). The current call sites all pass `record_event=False`, but the parameter is opt-in per-call and easy to flip. If anyone flips it on for a flag that gates a per-row feed render, the events table will grow at request-rate forever. The CRUD layer has no `DELETE FROM feature_flag_events WHERE timestamp < ?` job, no max-row-count trigger, and the schema (migrations/024 L70-78) has no TTL column. Disk-fill DoS via well-meaning observability. | **Fix:** add a daily aggregation job (`flag_eval_daily(flag_key, date, true_count, false_count)`) and a 14-day retention sweep that drops raw rows. Hard-cap `record_event=True` sites to a known allowlist enforced inside `features.is_feature_enabled` (`if record_event and flag_key not in features.METRIC_ENABLED_FLAGS: record_event = False`).

### LOW

- **LOW | features.py L117-142 | Default-deny on unknown flags is correct, but on DB-lookup failure the path returns `False` *silently* with only a `log.warning`.** Spec asked for "default-deny semantics on unknown flags" — `_evaluate(None, user)` returns False (L118-119), so an absent row defaults closed. Good. But L100-103 catches `Exception` on the DB read and returns False with `log.warning("feature flag lookup failed key=%s: %s", flag_key, exc)`. This is fail-closed (correct) but the log line is rate-limit-free, so a sqlite I/O storm during a snapshot would emit one warning per `is_feature_enabled` call — log-flood, not a security issue per se but obscures the storm by burying the root cause under thousands of warnings. | **Fix:** wrap with `log.warning(...)` only when `(time.time() - _last_warn) > 5`, or move the broad except into the caller. Otherwise, the default-deny semantic itself is sound — keep it.

- **LOW | admin_routes.py L78-93 | `_audit()` swallows every exception with `try/except: pass`.** Even when the AuditAction constants exist, any future schema-mismatch (e.g., audit_log adds a NOT NULL column without a default) silently drops the audit row. The principle "audit must never block the action" is right; the implementation should at minimum `log.warning("audit emit failed action=%s err=%s", action, exc)` so an ops dashboard surfaces the regression. Currently the only way you'd notice is by running a manual `SELECT COUNT(*) FROM audit_log WHERE action='feature_flag.update'` and noticing it's zero. | **Fix:** replace the bare `pass` with `log.warning("audit emit failed action=%s: %s", action, exc, exc_info=True)`.

- **LOW | admin_routes.py L302-316 vs queries/admin.py L555-557 | `subproduct_key` is not validated at the DB layer.** `_normalize_subproduct` (admin_routes.py L338) is the only choke-point. But the DB function `db.create_feature_flag(..., subproduct_key=...)` (queries/admin.py L589, L608) accepts any string — no `CHECK` constraint, no foreign key to a `subproducts` table. So if anyone bypasses the route layer (e.g., a future migration, a console script, a non-route handler), they can write `subproduct_key='../../etc/passwd'` or `subproduct_key='/* sql comment */'` and that string becomes a row that `flag_evaluate_api` can echo back to authed users. Belt-and-braces missing. | **Fix:** the `subproduct` slugs are an in-memory dict (`subproduct.SUBPRODUCTS`). Surface them as a SQL `CHECK (subproduct_key IS NULL OR subproduct_key IN ('voters', 'crypto', ...))` constraint, regenerated as part of the SUBPRODUCTS migration. Or migrate `subproduct_key` to a foreign key against a real `subproducts(slug)` table.

- **LOW | admin_routes.py L364-377 | `_csv` parser silently drops non-integer entries.** `enabled_user_ids="alice,42,bob"` parses to `[42]` — alice and bob vanish without any warning. Admin types a typo and never knows the row didn't ingest it. Not a security issue but the lack of feedback compounds the L532-562 partial-update problem. | **Fix:** if `_csv` skips any tokens, surface them back in the redirect query-string (`?warn=skipped:alice,bob`) or render an inline warning banner on the flag list page.

## Cross-cutting observations

- **The audit story is structurally broken — not just a missing constant.** The combination of (a) absent AuditAction enum values, (b) `_audit()` `except: pass`, (c) missing `before=` capture in `flag_save`, and (d) missing both `before` and `after` in `flag_delete` means there is currently no way to reconstruct flag-flip history from the WORM table. Even after the CRITICAL fix lands, finding HIGH-2 is needed before "who flipped what when" actually works end-to-end. Add an integration test that does CRUD on a flag and asserts a 3-row audit trail with non-null `before` and `after` JSON.
- **There is no flag registry — the entire flag namespace is admin-typeable.** Most production flag systems pin the keyspace to a code-level enum so a deploy ships both the new key and the code that reads it. This codebase has neither — `features.py` has no `KNOWN_FLAGS` and no decorator that registers flag keys at import time. Treating "flag exists in DB" as the sole source of truth means an admin typo (e.g., `voters-beta` instead of `voters_beta`) creates a dead row that no code reads, and an attacker (or fat-fingered admin) can pre-stage rows for keys that downstream code will eventually consume. HIGH-1 closes this.
- **`/api/flags/evaluate/{key}` is the wrong primitive.** It exists to let the admin UI test a flag against a user; it should not exist as a runtime gate. Production code already calls `features.is_feature_enabled` in-process. Either restrict to admin (HIGH-3) or delete the endpoint and surface "evaluate" only via the existing `/admin/flags/{key}` edit page with a "test against user_id" form input.
- **Default-deny IS correctly implemented at the evaluation layer** (`features.py` L118-119, L142 — unknown row returns False, exception returns False, anonymous never matches rollout). The CRUD layer is the weak link, not the evaluator.
- **Rate-limit gate on `_require_admin_user`** (server.py L5334-5340: 30 mutations / 5 min per admin email) covers `flag_create`, `flag_save`, `flag_delete` since they're POST. Reasonable bound.
- **CSRF protection** (`server.py` L1238-1268, CSRFMiddleware) covers all three POSTs — none are on `_CSRF_EXEMPT_POSTS` (L1105) and none match `_CSRF_EXEMPT_POST_PREFIXES`. Good.
- **The phantom `gateway/feature_routes.py` filename** is itself a finding. The audit spec asked for a file that doesn't exist; the actual surface lives at `gateway/admin_routes.py:296-593`. Either move the feature-flag block into its own file (`gateway/feature_routes.py`) to match operator expectations, or update `RUNBOOK.md` / `ARCHITECTURE.md` to note the co-location. Today the surface is invisible to anyone grepping by filename.
