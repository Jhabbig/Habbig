# Audit — `gateway/export_routes.py` (GDPR data-export)

Adversarial review focused on requestor identity check, export completeness across PII-bearing tables, background-job stale-auth, and signed/time-limited URL strength.

**File audited:** `gateway/export_routes.py` (441 lines) — the **only** GDPR-export module wired into `server.py` at line 6324. The richer `gateway/exports/generator.py` (987 lines, 51 tables) exists but is **dead code** — not imported anywhere except its own tests.

Supporting files reviewed:
- `gateway/queries/data_exports.py` (DB helpers for the export-request row)
- `gateway/exports/generator.py` (dead — but used as a reference for what *should* be in the export)
- `gateway/impersonation.py` (impersonation blocklist)
- `gateway/server.py` lines 1257–1370 (CSRF), 2057–2118 (`current_user`), 6321–6328 (route registration)
- `gateway/migrations/175_email_watermarks.py`, `070_watermark_seeds.py`, `072_security_events.py`, `110_shared_market_cards.py`, `112_shared_predictions.py` (PII tables NOT in the export bundle)

**Severity counts:** 3 CRITICAL · 4 HIGH · 5 MEDIUM · 3 LOW

Severity rubric:
- **CRITICAL** — auth bypass, cross-tenant data leak, forgeable signed URL, GDPR Art. 15 *materially* unmet.
- **HIGH** — exploitable in narrow conditions; reachable from authed user; data integrity / completeness gap that is plausibly the user's headline question.
- **MEDIUM** — defense-in-depth, race conditions bounded by quota, log/audit gaps.
- **LOW** — code quality with security implications; foot-guns waiting on a refactor.

## Top 3 (headline)

1. **CRITICAL · export-completeness — ~50 PII-bearing tables are silently omitted from the production GDPR export.** The wired module (`export_routes.py._build_zip`, L129–231) writes only 6 sections: `README.txt`, `account.json`, `subscriptions.json`, `predictions/user_predictions.{json,csv}`, optionally `predictions/saved.*`, and `metadata.json`. **The notifications section is gated on `hasattr(db, "list_user_notifications")` at L208 — that function does not exist anywhere in `db.py` or `queries/*.py`, so the notifications block is silently skipped on every export.** The README at L139–158 advertises six additional sections (markets/viewed, sources/followed, signal_search, notifications/history, activity/login_history, predictions) that the code does not produce — the ZIP arrives with the README falsely claiming completeness. The sister module `gateway/exports/generator.py` (dead code) enumerates the correct ~51-table surface (`_collect` at L207–656). Concrete tables that *do* have rows in production and *do* belong in a GDPR Art. 15 export but are missing today: `user_market_views`, `followed_sources`, `user_topics`, `intelligence_conversations`, `intelligence_messages`, `user_market_alerts`, `email_send_log`, `user_sessions`, `user_bet_history`, `api_keys`, `telegram_user_links`, `email_unsubscribes`, `backtests`, `feedback_submissions`, `gifted_subscriptions`, `user_positions`, `user_prediction_stats`, `user_trading_addon_settings`, `user_market_credentials`, `polymarket_connections`, `kalshi_connections`, `whale_watchlist`, `notifications`, `notification_preferences`, `push_subscriptions`, `engagement_events`, `user_onboarding`, `user_first_week_goals`, `changelog_seen`, `market_takes`, `take_votes`, `user_follows`, `collections`, `collection_follows`, `saved_views`, `webhook_subscriptions`, `feedback_items`, `feedback_votes`, `feedback_comments`, `cancellation_attempts`, `subscription_pauses`, `user_invite_tokens`, `referrals`, `affiliate_accounts`, `weekly_reports`, `discord_user_connections`, `telegram_connections`, `embed_widgets`, `data_export_requests`, `audit_log` (where `target_id=user_id`), `security_events`, `email_watermarks`, `watermark_seeds`, `shared_market_cards`, `shared_predictions`, `shared_source_cards`. A regulator audit (GDPR Art. 15, CCPA §1798.110) would treat this as a Subject Access Request fulfilment failure — the user has data they aren't being given, *and* the README lies about what they're getting. **Fix:** wire `exports/generator.py.generate()` into `_enqueue_export` (it already speaks the same `data_export_requests` row contract via `db.update_export_status` — small adapter at `_run_export` L234). Delete the truncated `_build_zip` in `export_routes.py`. Add an explicit-allowlist test that asserts every table in `gateway/migrations/*.py` with a `user_id` column is either present in `_collect` or appears in a documented exclusion list with a redaction rationale (compare against `_BLOCKED_PATTERNS`-style allowlist).

2. **CRITICAL · signing-key fallback — `_export_secret()` derives a publicly-guessable HMAC key in any deployment that forgot to set `GATEWAY_COOKIE_SECRET` or `DATA_EXPORT_SIGNING_KEY`.** L75–80: `raw = os.environ.get("GATEWAY_COOKIE_SECRET") or os.environ.get("DATA_EXPORT_SIGNING_KEY") or f"dataexport:{EXPORT_DIR}"`. The fallback string `f"dataexport:{EXPORT_DIR}"` resolves to either `"dataexport:/tmp/narve-exports"` (default env) or whatever value an attacker can read from a leaked `Dockerfile`, `systemd` unit, or `/proc/self/environ` of a co-tenant container. Combined with the `_sign(export_id, user_id, expires_at) → hmac.sha256[:32]` shape at L83–85 and the fact that all three inputs are knowable (numeric `export_id` is sequential, `user_id` is the victim's, `expires_at` is `requested_at + 7d`), an attacker can **forge** signed download URLs for arbitrary `(export_id, user_id)` pairs and pull any user's GDPR ZIP from anywhere on the public internet — no session, no cookie, no CSRF. The sister `exports/generator.py._signing_secret()` (L73–99) correctly `raise RuntimeError` on missing secret; `export_routes.py` chose silently-fail-open. Worse, there is no startup assertion that either env var is set, so the fallback is the *operational default* during dev/test, and the fallback string can plausibly survive into production if a deploy script forgets to inject the secret (a recent precedent: see `NARVE_SECURITY_AUDIT.md` L1435). **Fix:** copy the `RuntimeError` from `exports/generator.py` — if neither env var is set, refuse to start the route (or refuse to mint any URL). Add a `pytest` that boots the app with both env vars unset and asserts `/api/account/export` returns 500. Separately: stop truncating HMAC to 32 hex chars — use the full 64 chars (256 bits); the 128-bit truncated form is still adequate against birthday attacks but gives an attacker an unnecessary 128-bit head-start vs. the longer fallback secret material.

3. **CRITICAL · impersonation can export the target user's data.** `gateway/impersonation.py._BLOCKED_PATTERNS` (L45–83) blocks `/account/password`, `/account/email`, `/account/2fa`, `/billing`, `/admin`, `/intelligence`, etc. — but **does not list `/api/account/export`, `/api/account/exports`, `/api/account/export/.../download`, or `/settings/privacy`**. The flow: an admin starts `narve_impersonation` for victim user_id `V`. `current_user()` (`server.py:2057–2098`) returns the **target** row, so `api_request_export` at `export_routes.py:279` sees `user_id=V`. The CSRF gate passes (impersonation cookie ≠ CSRF cookie, both are sent), `db.create_data_export_request(V)` writes a row owned by `V`, the ZIP is built from `V`'s data, the `download_url` is rendered HMAC-signed for `(export_id, V, expires_at)`, and the admin downloads it. The audit trail is split: `impersonation_actions` records the HTTP path (so post-hoc detection is possible) but `audit_log` has **no** entry for "GDPR export requested" — the route never calls `security.audit.log_action()`. A malicious admin can exfiltrate any user's bundle in under a minute and the only ledger is an HTTP-path row that says nothing about export contents. **Fix:** add the four paths above to `_BLOCKED_PATTERNS` *and* to `_READ_ALSO_BLOCKED_PATTERNS` (the download URL is a GET). Belt-and-braces: at the top of `api_request_export`, `api_list_exports`, and `api_download_export`, check `getattr(request.state, "impersonation", None)` and 403 if present, regardless of the blocklist (the blocklist is an HTTP-path gate; an in-handler check survives any future re-mapping). Add `audit.log_action(AuditAction.DATA_EXPORT_REQUESTED, target_user_id=user_id, ...)` at L292 and `DATA_EXPORT_DOWNLOADED` at L348 so the audit_log has a first-class record of every export request regardless of who triggered it.

## Findings (severity-sorted)

### CRITICAL

- **CRIT-1 — Export completeness (~50 tables missing).** See Top-3 #1. Lines: `export_routes.py:129–231`, README at L139–158, `hasattr(db, "list_user_notifications")` at L208.

- **CRIT-2 — Forgeable signed URL via env-var fallback.** See Top-3 #2. Lines: `export_routes.py:71–80`, `_sign` L83–85, `_verify` L88–90, `_download_url_for` L93–95.

- **CRIT-3 — Impersonation exfiltrates target's GDPR bundle.** See Top-3 #3. Lines: `impersonation.py:45–83`, `server.py:2057–2098`, `export_routes.py:277–352`.

### HIGH

- **HIGH-1 · L38, L136 | World-readable export directory.** `EXPORT_DIR = Path(os.environ.get("DATA_EXPORT_DIR", "/tmp/narve-exports"))` — on the default value or any operator-set value that isn't pre-chmod'd, `_build_zip` calls `zip_path.parent.mkdir(parents=True, exist_ok=True)` (L136) with **no `mode=0o700` and no follow-up `chmod`**. On a shared host (or a single-tenant VM with multiple non-root daemons), any local user can `ls /tmp/narve-exports/` and read every export. The sister `exports/generator.py:60–64` correctly sets `mode=0o700` and `os.chmod(..., 0o700)`; `export_routes.py` does not. **Fix:** at module init, `EXPORT_DIR.mkdir(parents=True, exist_ok=True, mode=0o700); os.chmod(EXPORT_DIR, 0o700)` — copy the block from `exports/generator.py:60–64` verbatim. Also chmod the resulting ZIP file to `0o600` after writing it at L138, so an attacker who controls the directory umask but not the file still can't read individual ZIPs.

- **HIGH-2 · L284–290 | Failed exports count against the user's 24h quota — self-DoS.** `db.last_user_data_export_ts` (`queries/data_exports.py:56–64`) returns the most recent `requested_at` **regardless of `status`**. So if a user's export crashes (status='failed' at L263) one minute after they request it, they are locked out for the next 23h59m. Worse, the error message at L289 (`f"Rate limit: next export available in {retry_in // 3600}h"`) integer-divides retry_in by 3600 — a user 59 minutes away from re-eligibility sees "0h" and reasonably retries, hitting the rate limit again. The user has no recovery path other than contacting support. **Fix:** filter `last_user_data_export_ts` by `status IN ('pending','processing','ready')` — only successful or in-flight requests should burn quota. Failed requests should be immediately retryable. Also: round-up display (`(retry_in + 3599) // 3600`) so "0h" never appears.

- **HIGH-3 · L322–352 | Download URL doubles as the bearer token — leaks via referer, browser history, screenshare, server logs.** `api_download_export` (L322) explicitly states "does NOT require session auth" and authenticates purely from query-string parameters `u`, `e`, `s`. Anywhere the URL appears — Cloudflare access logs, the user's browser history, a screenshare of `/settings/privacy`, a copy-paste into a chat, the Referer header if the user clicks an outbound link from the download page — that URL is a fully-functional 7-day download capability for the user's complete data bundle. The page at `privacy_settings_page` (L355–425) renders the same URL into an `<a href=...>` (L370), so any third-party script on the page (or future XSS) can read `document.querySelector` and exfiltrate. **Fix:** require BOTH the HMAC signature AND a valid session cookie at download time. Drop the "shareable from notification email" comment at L323–324 — the actual `_run_export` doesn't send email (HIGH-4 below), so the shareable-link rationale is moot. If a future email integration is added, switch the email to send a *redirect URL* gated on session, not a direct file URL. Belt-and-braces: add `Referrer-Policy: no-referrer` to the download response, and `Cache-Control: private, no-store` so the ZIP isn't cached by intermediaries.

- **HIGH-4 · L355–425 + L234–266 | UI promises an email; backend never sends one — user is stuck polling a page with no auto-refresh.** The privacy-settings page renders "*No exports yet. Request one below — we'll email you when it's ready.*" at L388. But `_run_export` (L234–266) never calls into `jobs.email_jobs` or `email.send_template` — only `exports/generator.py:953–977` does, and that module is dead code. The user gets a UI claim that the system will email them, the system does not email them, the page does not auto-poll (no `setTimeout`, no JS to refetch `/api/account/exports`), and there's no inline JS handler on the request button (the request button itself isn't even rendered in this excerpt — the page expects template-side wiring that isn't shown to be present). This is a user-trust / dark-pattern issue: the consent flow advertises a notification commitment the code doesn't honor. **Fix:** either (a) actually send the "your export is ready" email — port `exports/generator.py:953–977` into `_run_export`'s `try:` block, or (b) edit the UI copy at L388 to "*Refresh this page to check status; no email is sent.*" and add a `<meta http-equiv="refresh" content="15">` while any row is in `pending`/`processing`. (a) is the right answer for GDPR — Art. 15(3) says "without undue delay" and the regulator expects async exports to notify when done.

### MEDIUM

- **MED-1 · L284–292 | TOCTOU race on rate-limit check.** Two simultaneous POSTs from the same user both call `last_user_data_export_ts(V)`, both see `None` (or both see the same stale value), both insert a new `data_export_requests` row, both enqueue a ZIP build. The DB has no unique constraint on `(user_id, requested_at)` and no transactional COUNT-then-INSERT. Cost: wasted thread-pool slots (3 workers serve all users); a user with a script can DoS the pool by spawning N concurrent requests from N tabs. **Fix:** wrap the check-and-insert in a single `BEGIN IMMEDIATE` transaction, or add a partial unique index `CREATE UNIQUE INDEX idx_data_export_one_pending ON data_export_requests(user_id) WHERE status IN ('pending','processing')` — the second request then `IntegrityError`s and 429s cleanly. Cheaper still: an in-process `dict[user_id, last_request_ts]` set just before `create_data_export_request`, double-check after.

- **MED-2 · L234–266 | Thread-pool jobs lost on worker restart — stale `processing` rows forever.** `_executor = ThreadPoolExecutor(max_workers=3)` (L46) is **in-process**, no persistence. If a gunicorn worker is killed mid-export (deploy, OOM, segfault), the row stays `status='processing'` forever — and because of HIGH-2 it now blocks the user from requesting another export for 24h *and* there's no UI hint that the job is dead. **Fix:** add a startup hook that finds `data_export_requests WHERE status IN ('pending','processing') AND requested_at < now() - 10*60` and either re-enqueues them or marks them `failed` with a clear `error` ("worker restart"). Long-term: move the work to an ARQ/Redis worker (the dead `exports/generator.py:906` already returns an ARQ-compatible dict). At minimum, log a warning on graceful shutdown listing in-flight `request_id`s so post-mortem is possible.

- **MED-3 · L234–266 | No re-auth check in worker — deleted users can still produce exports.** The job re-reads the DB row at L237 and gets `user_id`, then calls `db.get_user_by_id(user_id)` at L162 inside `_build_zip`. If the user has been deleted between request-time and job-time (account closure, admin removal), `u` is `None`, `account = {}`, and the resulting ZIP has an empty account.json + whatever cascade-not-yet-applied rows exist for orphan `user_id`. Not a security issue *today* (an orphaned ZIP contains nothing exploitable) but the row is still written to `data_export_requests` with `status='ready'` and a signed download URL — anyone with the URL (CRIT-2, HIGH-3) gets the (mostly empty) ZIP after the account is gone. **Fix:** at the top of `_run_export`, re-check `db.get_user_by_id(row["user_id"])` and bail with `status='failed'` + `error='account deleted'` if missing. Also delete the partially-written ZIP file on failure (currently only marks the row).

- **MED-4 · L292, L348 | No `audit_log` entry for export request or download.** Neither `api_request_export` nor `api_download_export` calls `security.audit.log_action()`. The DB row `data_export_requests` is the only ledger. This means:
  - A user whose data was exported has **no entry in their own `audit_log` table** when reading it back later (compare to the `_scrub_audit_row` path in `exports/generator.py:644–654`).
  - An attacker who pivots into the gateway DB and *deletes* the `data_export_requests` row (or `UPDATE`s `download_url` to a forged one) leaves no second source of truth.
  - The impersonation-actions table (`server.py:1547`) only records the HTTP path, not the `export_id`/`download_url`/`file_size` — forensics post-incident is harder than it should be.
  
  **Fix:** add `audit.log_action(AuditAction.DATA_EXPORT_REQUESTED, target_user_id=user_id, target_description=f"export_id={request_id}", request=request)` at L292, and `DATA_EXPORT_DOWNLOADED` at L348 with the file size + IP from `request.client.host`. Wire both `AuditAction` enum entries if they don't exist (`security/audit.py`).

- **MED-5 · L165–167 | `_build_zip` redaction list is incomplete vs. the live schema.** L165–167 drops `password_hash`, `password_salt`, `totp_secret`, `backup_codes`, `pending_totp_secret`. Missing from the redaction list: `stripe_customer_id` (PII leaked to user, but if export is leaked an attacker can correlate with Stripe webhooks), `referred_by_user_id` (other user's id — that's their PII, not the exporter's), `last_login_ip` / `last_login_user_agent` columns if present (debate-able: it's *their* IP, fine to export to them; concern is post-leak forensic value to an attacker), and `api_key_pepper`/`signing_key`-style columns if any have been added to `users` since this list was written. The sister `exports/generator.py:_scrub_user_row` (L158–161) is **even worse** (drops only `password_hash` and `password_salt`). **Fix:** invert the policy — define an **allowlist** of `users` columns that may be exported, default to drop. Add a test that asserts every column in `PRAGMA table_info(users)` is either in the allowlist or in a documented redaction list with a rationale.

### LOW

- **LOW-1 · L38, L243 | `file_path` stored in DB is trusted at download time without prefix check.** L343 reads `row["file_path"]` and passes it to `FileResponse(path=str(file_path), ...)` after only `is_file()` validation. If a separate vulnerability (admin UI, SQL injection, ORM bug) ever allowed an attacker to `UPDATE data_export_requests SET file_path='/etc/passwd' WHERE id=...`, the next download serves that file. **Fix:** assert `file_path.resolve().is_relative_to(EXPORT_DIR.resolve())` before serving. Python ≥3.9 has `PurePath.is_relative_to`; or use `os.path.commonpath` for older runtimes. Defense-in-depth; no live exploit today.

- **LOW-2 · L46, L271 | `ThreadPoolExecutor(max_workers=3)` is a global DoS vector via legitimate users.** Three slots serve **all** users globally. If three users each kick off a long-running export (large `intelligence_messages` history, multi-GB bet_history), a fourth user's request sits in the executor's unbounded queue indefinitely — and the user's `data_export_requests` row stays at `status='pending'` with no UI indication. Combined with MED-1 (race), a single user with three tabs can occupy all three slots. **Fix:** size the pool to `max(3, os.cpu_count())`, add a bounded `queue.Queue(maxsize=10)` in front of `_enqueue_export` that returns 503 with Retry-After if full, and surface a "queued, position N" hint in `api_list_exports` (`pending` rows ordered by `requested_at`, position = index). Or — preferred — punt to ARQ as MED-2 suggests.

- **LOW-3 · L100–113 | `_rows_to_list` swallows malformed rows silently.** `try / except / pass` at L107–112 means a row that fails dict conversion is silently dropped from the export. For a GDPR export, "silent drop" is the wrong default — the user is being told this is their complete data, and a single bad row could come from a column rename mid-deploy. **Fix:** log a `WARNING` with the failing row's primary key, and either fail-loud (status='failed' with a recoverable error) or write a `_errors.json` into the ZIP listing every dropped table/row pair so the user knows what's missing.

## Non-findings (verified clean)

- **CSRF on `/api/account/export` POST:** `CSRFMiddleware` (`server.py:1257–1370`) enforces double-submit-cookie on every POST except a documented exemption list (`_CSRF_EXEMPT_POSTS`). `/api/account/export` is not in any exempt list — POST requests must carry both `_csrf` cookie and the matching `x-csrf-token` header (or form field). Verified by reading the middleware.
- **HMAC constant-time compare:** `_verify` (L88–90) uses `hmac.compare_digest` — no timing side-channel on the signature check itself. The truncated-128-bit signature length is the only weakness (acceptable for a 7-day URL with rate-limited issuance).
- **`expires_at` enforcement:** L333 checks `int(time.time()) > expires_at` before HMAC verification, so an expired URL fails cleanly with 410. Also checked: integer-coercion of query params catches alphabetic input via `ValueError → 400` at L329–330. `expires_at` is bound by HMAC, so a future-dated `e=` query param is rejected by the signature check unless the attacker has the secret (see CRIT-2 for that path).
- **Ownership at download:** L339 asserts `row["user_id"] != user_id` against the HMAC-bound `user_id`. An attacker swapping `?u=` to point at another user's export forces HMAC re-verification with the new user_id; without the secret, the sig won't match.
- **Account-row password scrub:** L165–167 strips `password_hash`, `password_salt`, `totp_secret`, `backup_codes`, `pending_totp_secret`. See MED-5 for the (small) gaps.
- **Path-traversal in `file_path` write:** `_run_export` builds the path as `EXPORT_DIR / f"export_{request_id}_u{user_id}.zip"` (L243) — both ints, no string injection. See LOW-1 for the defensive-read concern.
- **`current_user` returns dict (not `sqlite3.Row`):** L279 reads `user["user_id"]` — guaranteed key-access works because `current_user` in `server.py:2057` always builds a dict. No `.get()` foot-guns reachable here.
- **Stripe / billing surface not entangled with export:** the export bundle includes `subscriptions` table rows but does not include raw Stripe webhook payloads or PaymentMethod fingerprints — those live in `processed_stripe_events` and are correctly *not* user-PII (they're Stripe-side identifiers).
