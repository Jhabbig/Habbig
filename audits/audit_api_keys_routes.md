# Audit — `gateway/api_keys_routes.py`

Adversarial review focused on auth guards, CSRF, SQLi, XSS, PII, race conditions, audit gaps, rate limits, ownership checks.

**File audited:** `gateway/api_keys_routes.py` (430 lines, reviewed at HEAD `8689ea5`).

**Severity counts:** 1 CRITICAL · 2 HIGH · 4 MEDIUM · 3 LOW

Findings are sorted strictest-first. Severity rubric:
- **CRITICAL** — auth bypass, data leak across tenants, RCE, secret exposure.
- **HIGH** — exploitable in narrow conditions; reachable from authed user; data integrity.
- **MEDIUM** — defense-in-depth, race conditions bounded by quota, log hygiene.
- **LOW** — code-quality with security implications; foot-guns waiting on a refactor.

---

## Top-10 findings

- **CRITICAL | L323-325 | `admin_api_keys_page` does NOT reject non-admin users — the guard check is structurally wrong, so the cross-tenant key listing fires for *any* authenticated user. `_require_admin_user(request, page=True)` returns `None` for non-admin (per its docstring at `server.py:5176`), not a `RedirectResponse`. The handler then runs `if hasattr(admin, "status_code"): return admin` — `None` has no `status_code`, so flow continues to `q_api_keys.list_all_api_keys()` at L327 (loading every key for every user into memory) before crashing at L387 only because `_role_badge(None)` happens to raise `AttributeError → 500`. The auth correctness depends on an accidental crash in a downstream helper. Any refactor that wraps `_role_badge` in `try/except`, makes it None-safe, or moves it after the template render will silently turn this into a full cross-tenant disclosure of every key (`key_prefix`, owner email, scopes, origins, usage_count, timestamps). | **Fix:** after L323 replace the guard with `if not admin or hasattr(admin, "status_code"): return admin or _denied_response(request)`. Apply the same fix to `admin_api_keys_revoke` at L394-396 (same bug, lower blast radius because the path needs both POST + CSRF + the rate-limited admin-mutation gate, but `q_api_keys.admin_revoke_api_key(int(key_id))` would still execute against arbitrary `key_id` from any authed user if a future refactor papers over the crash).

- **HIGH | L238 + L253 | TOCTOU race on per-tier key quota — two concurrent POSTs from the same user can both pass `_active_key_count(...) >= quota["max_keys"]` and both mint a key, exceeding the tier cap (e.g., a free-tier user gets 2 keys when capped at 1). `_active_key_count` reads outside any transaction, then `create_api_key` inserts in a separate connection (`queries/api_keys.py:136`). Free/trader users on the 1-key limit are the easy target; pro at 5 is also reachable. Quota bypass leads to multiplied rate-limit budget (each minted key has its own `rate_limit_hour`), which is the actual business risk. | **Fix:** enforce the quota at SQL-write time. Add a partial unique index `CREATE UNIQUE INDEX idx_api_keys_one_per_user ON api_keys(user_id) WHERE revoked_at IS NULL AND tier='embed'` for free/trader, or wrap count+insert in a single `BEGIN IMMEDIATE` transaction. Cheaper alternative: `INSERT … WHERE (SELECT COUNT(*) FROM api_keys WHERE user_id=? AND revoked_at IS NULL) < ?` and check `cursor.rowcount`.

- **HIGH | L362 | XSS in admin key listing via attacker-controlled `owner_email`. The owner email comes from the `users` table via `list_all_api_keys` join. It is escaped *inside the link text* (`owner_safe` at L338, L363) and *inside the confirm dialog* (L348), but L362 puts the **un-escaped raw `owner`** into the `href` attribute: `f'<a href="/admin/users?q={html.escape(owner)}"`. Wait — `html.escape` IS applied here, but `html.escape` does not encode the URL-significant characters `&`, `?`, `=`, `#`, or unicode. A registered email containing `"><script>` would be escaped (`&quot;&gt;...`), but an email containing `javascript:` won't be — and more importantly, `html.escape` is not a URL-encoder. The query-string injection (`q=foo&admin_action=delete`) is achievable if an email like `foo@bar.com&drop=1` ever lands. Email validation upstream usually blocks `<>`, but plus-aliases, IDN, and quoted-string emails (RFC-5322 permits `"a<b>c"@example.com`) can slip through. | **Fix:** wrap `owner` in `urllib.parse.quote(owner, safe='@')` before interpolation, then `html.escape` for the attribute context. Also `html.escape(..., quote=True)` (the default) is what's used, which is fine for the value — the bug is missing URL-encoding, not HTML-escaping.

- **MEDIUM | L271 | Audit log persists a fragment of the raw API key (`target_id=raw_key[:12]`). `raw_key` is `nv_emb_<32hex>`, so the first 12 chars are `nv_emb_<5hex>` — only 5 hex (20 bits) of secret material, the rest 108 bits stay secret. Not directly exploitable today, but: (a) inconsistent with `api_key.revoke` audit at L306 which uses the numeric `key_id` — pivoting between audit entries for the same key is harder than it should be; (b) if an admin viewing audit_log accidentally treats `target_id` as "safe to share" (e.g., posting it in a Slack channel) they leak the public prefix that is supposed to be derivable only from the DB row. The `api_keys.key_prefix` column already stores exactly this — duplicating it in audit_log gives nothing. | **Fix:** mirror the revoke audit shape — emit `target_id=<row_id_after_insert>` (return the inserted id from `create_api_key`, which currently returns `(raw_key, key_hash)` — change to `(raw_key, key_hash, row_id)`).

- **MEDIUM | L262 | `log.exception("create_api_key failed user=%s: %s", user["user_id"], exc)` — the exception chain may carry the SQL statement and parameters under `sqlite3.IntegrityError`, which means the inserted `key_hash` (SHA-256 of the raw key) ends up in stdout/file logs. The hash is not the secret, but anyone with log access can confirm guessed keys against `_hash_key(guess) == logged_hash`. Combined with the 5-hex prefix in audit_log this drops the brute-force cost. | **Fix:** swap `log.exception(...)` for `log.error("create_api_key failed user=%s err_type=%s", user["user_id"], type(exc).__name__)` — no `exc` arg, no `.exception()` traceback. Keep the stack trace gated behind `log.debug` for ops.

- **MEDIUM | L225-292 + L295-311 | No per-user rate limit on `api_keys_create` or `api_keys_revoke`. CSRF middleware blocks naive cross-origin requests, but a phished session cookie or a same-origin XSS (e.g., from a different route on `narve.ai`) can drive create→revoke→create churn. Each create + revoke pair writes audit rows and updates the DB; at the create endpoint it also burns the user's `_active_key_count` budget so legitimate use temporarily fails with 409. The admin-side revoke is rate-limited (`server.py:5196-5202`, 30 mutations / 5 min per admin email) but user-side write paths in this file are not. | **Fix:** add `_is_rate_limited(f"api_key_mut:{user['user_id']}", 20, 300)` at the top of both `api_keys_create` and `api_keys_revoke`, returning 429.

- **MEDIUM | L273 | Audit `notes` field interpolates user-supplied `origins_raw` verbatim (`notes=f"scopes={scopes} tier={tier} origins={origins_raw or 'any'}"`). `origins_raw` is read at L232 with no length cap (only `name` is `[:80]`-truncated at L230). A user posting `allowed_origins=` of arbitrary size bloats the audit_log row — DoS via storage growth — and can inject control characters that break log-grep tooling or terminal renderers viewing the audit page. | **Fix:** truncate at logging time: `origins_for_log = (origins_raw or 'any')[:200]` and strip control chars: `''.join(c for c in s if c.isprintable() or c == ' ')`.

- **LOW | L43-49 | `_TIER_QUOTAS["none"]` exists with `max_keys=0` but `_quota_for(tier)` at L86-90 falls back to `_TIER_QUOTAS["free"]` (1 key) on unknown tier. If `db.get_user_subscription_tier(user_id)` ever returns an empty string, a literal `None`, or a tier the DB knows but the code doesn't, the user gets the free quota instead of being denied. Belt-and-braces: paid plans have a clear quota entry, but an enterprise-tier user whose subscription expires mid-session and is now in `none` state should NOT be granted free-tier minting. | **Fix:** make `_quota_for` default to `_TIER_QUOTAS["none"]` (deny) instead of `_TIER_QUOTAS["free"]`. This is fail-closed; the current code is fail-open.

- **LOW | L286 | `raw_key=raw_key` passes the freshly-minted key through `render_page` as a `raw_`-prefixed context value, which `server.py:2547` interpolates **without HTML escaping**. The minted key is currently `nv_emb_<32 hex chars>`, none of which need escaping, so there's no live XSS. But the parameter name signals "this is HTML, trust it" to the template engine, and a future change to `_mint_raw_key()` that broadens the alphabet (e.g., base64 with `+`/`/`/`=`) would silently introduce an XSS sink on the reveal page. | **Fix:** rename to `key=raw_key` (no `raw_` prefix → template engine escapes it) and update `settings_api_key_reveal.html` placeholder to match. Two-line change; eliminates the foot-gun permanently.

- **LOW | L332-333, L337 | Mixed access pattern on `sqlite3.Row` — the admin handler uses defensive `r["k"] if "k" in r.keys() else default` for `usage_count`, `allowed_origins`, `owner_email`, but the user-facing handler at L150-157 uses direct subscript with no fallback. `list_api_keys` (`queries/api_keys.py:230`) and `list_all_api_keys` (`queries/api_keys.py:297`) both `SELECT` these columns explicitly, so the defensive check is dead code today — but the inconsistency hints at a half-finished migration. If a future schema change drops a column from one query but not the other, the user-facing page 500s while the admin page silently renders. Tests would not catch this because both currently always return all columns. | **Fix:** drop the `if "k" in r.keys() else default` defensiveness, or apply it uniformly. Pick one. Also note: `sqlite3.Row` has no `.get()` method (you can't write `r.get("usage_count", 0)`) — the audit didn't find a `.get()` call here but if anyone adds one it will `AttributeError` in production.

## Non-findings (verified clean)

- **CSRF on state-mutating routes:** `CSRFMiddleware` (`server.py:1218`) enforces double-submit cookie on every POST except a documented exemption list (`_CSRF_EXEMPT_POSTS`, `_CSRF_EXEMPT_POST_PREFIXES`). `/settings/api-keys`, `/settings/api-keys/{id}/revoke`, `/admin/api-keys/{id}/revoke` are NOT exempt — POST forms get CSRF enforcement (form field auto-injected by `render_page` at `server.py:2551-2558`).
- **SQL injection via f-strings:** none. All DB writes go through parameterized queries in `queries/api_keys.py` and `db.py`. f-strings only build HTML (with `html.escape`).
- **`key_id: int` path param:** typed; FastAPI rejects non-numeric input with 422 before the handler runs.
- **`html.escape` coverage in HTML row builders:** `_origin_badges`, `_scope_badges`, `key_prefix`, `name`, `owner` (in link text and confirm dialog) are all escaped. The single gap is the `href` attribute URL-encoding (HIGH #3 above).
- **Raw key persistence:** the file does NOT persist the raw key. `create_api_key` returns `(raw_key, hash)`; only the hash is INSERTed (`queries/api_keys.py:141-148`); the raw is rendered to the reveal page and dropped.
- **Hardcoded creds:** none.
- **Ownership scoping on user-side revoke:** `q_api_keys.revoke_api_key(key_id, user["user_id"])` (`queries/api_keys.py:240-254`) constrains the UPDATE by `user_id`, so user A cannot revoke user B's key by guessing key_id. Verified.
